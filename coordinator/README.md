# Coordinator — a parallel "OS" of sub-agents for chemistry-astar

This directory turns the `chemistry-astar` repo into a multi-agent operating
system: one **supervisor** schedules several **worker** agents that build and feed
the study app over the 21-day challenge. It replaces the original generic
coordinator prompt with one wired to *this* codebase.

## The mental model

| OS concept        | Here                                                          |
|-------------------|---------------------------------------------------------------|
| Kernel / scheduler| `agents/supervisor.md`                                        |
| Processes         | explainer, question-generator, mock-runner, validator, feature-builder |
| Shared memory     | `coordinator/state.json` (only mutable coordination state)    |
| Read-only sysfiles| `src/data/spec.js`, `src/data/plan.js`, `src/data/paperIndex.json`, `smartmark/topics.py` |
| Syscall contract  | the JSON schemas in `src/api/anthropic.js`                    |
| Disk / persistence| git                                                           |
| Boot config       | `coordinator/config.json` + `coordinator/CLAUDE.md`           |

## Two layers — the thing the original prompt missed

- **Build/ops layer = these agents.** Claude Code instances that generate content,
  QA it, edit code, and commit. They authenticate through Claude Code.
- **Runtime layer = the app itself.** `src/api/anthropic.js` calls Claude live in
  the browser for the student.

The build layer's payoff: the deployed app exposes its API key in the bundle
(root `README.md` → Security). So instead of relying on risky live calls, the
agents **pre-generate** a content pack (notes + question banks) that the app
imports into `localStorage`. The student gets instant, key-free study material,
and the strict marking still happens — either offline against model answers, or
locally in SmartMark. Pre-generation, not live calls, is the default.

## Folder layout

```
coordinator/
  CLAUDE.md            shared context every agent reads first
  config.json          paths, models, agent contracts, content-pack format
  state.json           live coordination state (supervisor owns this)
  schedule.json        21 days, derived from src/data/plan.js + agent assignments
  dashboard.md         (generated) human-readable status
  agents/              copy-paste prompts: supervisor + 5 workers
  tasks/               (generated) day-{N}.json task packets
  qa/                  (generated) validator verdicts per spec id
  results/             (generated) mock scores + topic breakdowns

content/               (generated by workers; safe to commit — originals only)
  notes/{specId}.json      explainer output (generateLesson shape)
  questions/{specId}.json  question-generator output (generateQuestions shape)
  mocks/{date}.json        mock-runner assembled papers
  pack.json                merged importable pack (built by feature-builder)
```

`content/` holds only original, OCR-style material. Real past-paper PDFs and mark
schemes never leave `smartmark/papers/` (git-ignored).

## Daily workflow

Each agent is one Claude Code tab (or sub-agent). The supervisor lives in its own
tab; you spin up workers per the day's plan.

**Morning — "Run Day N"** (in the supervisor tab):
1. Supervisor computes the real day from `START_DATE`, loads `schedule.json`,
   reads your `imports/localStorage-export.json` if present, and writes
   `tasks/day-{N}.json`.
2. It prints which agents to launch and on which spec points.
3. You open a tab per worker and paste its prompt from `agents/`. Each worker
   reads its task from `tasks/day-{N}.json` and writes to `content/…`.
4. The validator QAs the new content and writes `qa/*.json`.

**Evening — "Complete Day N"** (back in the supervisor tab):
5. Supervisor collects outputs, bounces anything the validator failed, updates
   `state.json` + `dashboard.md`, and commits.

Review days (7, 14, 21) swap the learning workers for the mock-runner: it
assembles a mock from the banks, the validator marks it, and weak topics (<70%)
are written back to `state.json` for the next learning cycle to target.

## Quick start

```bash
# from repo root
cp -r coordinator .                 # if not already here
mkdir -p content/notes content/questions content/mocks
```

Then, in your supervisor tab:

> Read coordinator/CLAUDE.md and coordinator/agents/supervisor.md. Run Day 7.

(Today is Day 7 of the challenge — a Module-2 review/mock day, per the real
`START_DATE` of 2026-06-16. `state.json` is already seeded there.)

To personalise scheduling, first export your app data: build feature #4 in
`agents/feature-builder.md`, export to `imports/localStorage-export.json`, and the
supervisor will prioritise your weak/low-confidence spec points automatically.

## What changed vs. the original coordinator prompt

- **No duplicated curriculum.** The original recreated modules, a 21-day schedule,
  and progress trackers from scratch. Here the schedule is *derived from the real
  `plan.js`*, and spec/topic/paper data are referenced, never forked — so they
  can't drift.
- **Schema-locked output.** Every worker emits the exact JSON shapes the app's
  `anthropic.js` already uses (`generateLesson`, `generateQuestions`,
  `markAnswers`, `paperScores`), so output is drop-in.
- **Correct facts.** Board fixed to OCR A H032 (not AQA); real spec ids
  (`2.1.3b`, …); real model split (`sonnet-4-6` / `haiku-4-5`); real localStorage
  keys; 46 spec points, not the placeholder 60.
- **Copyright kept clean.** Original questions only in git; real papers stay in
  SmartMark; the validator actively spot-checks for lifted past-paper text.
- **A real payoff.** The agents exist to pre-generate a key-free content pack and
  to harden the app — not just to shuffle JSON between folders.
